Liquibook은 유연합니다. 곡예사처럼 유연합니다

Liquibook은 다양한 사용 사례를 충족하도록 설계되었습니다.

이는 여러분이 다음을 선택할 수 있음을 의미합니다:

  • 주문 클래스
  • 포인터 선호도 (스마트 또는 표준)
  • 스레딩 모델
  • 식별자
  • 다운스트림 처리
  • 애드온

어떻게 그렇게 유연한가요?

여러분의 주문 클래스가 사용되며, 사소한 인터페이스만 구현하면 됩니다. 없는 경우 샘플이 포함되어 있습니다.

OrderBook 클래스는 템플릿으로, 포인터 타입을 매개변수로 받습니다. -> 연산자를 사용하여 역참조하는 모든 포인터 타입을 사용할 수 있습니다.

Liquibook은 애플리케이션이 아니라 라이브러리이며, 여러분의 애플리케이션 내에서 사용할 수 있습니다. 따라서 동기화나 백그라운드 스레드가 없습니다. 다중 스레드를 가질지, 어떻게 동기화할지는 여러분이 선택합니다.

Liquibook은 주문 ID, 사용자 ID, 계정 ID, 또는 심볼을 사용하지 않고 오직 Order 포인터만 사용합니다. 주문 ID에서 포인터로의 매핑은 Liquibook 외부에서 수행됩니다.

Liquibook은 또한 주문 장부에 제공된 모든 주문이 해당 주문 장부의 심볼에 대한 것이라고 가정하며, 심볼에서 주문 장부로의 매핑은 Liquibook 외부에서 수행됩니다.

Liquibook은 거래나 호가로 인한 피드 게시와 같은 다운스트림 처리를 수행하지 않습니다. 결국 그것은 피드별로 다릅니다. 대신 Liquibook은 콜백을 대기열에 넣고, 여러분이 코드에서 이를 처리합니다.

Liquibook은 깊이/BBO 추적을 위한 선택적 애드온 클래스를 포함하며, 이는 콜백 처리에서 쉽게 통합될 수 있습니다. 예제는 impl 디렉토리의 SimpleOrderBook을 참조하십시오.
